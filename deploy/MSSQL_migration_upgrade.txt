MSSQL + Flask-Migrate — fixing error 3726 on ``flask db upgrade``
================================================================

Symptom
-------
``Could not drop object 'invoices' because it is referenced by a FOREIGN KEY constraint (3726)``

Cause
-----
Autogenerate produced a revision that drops ``invoices`` (and related tables) before every
foreign key pointing *to* those tables has been dropped (for example ``contributions.invoice_id``
→ ``invoices``).

Do **not** delete the ``migrations/`` folder or run ``flask db init`` on a database that
already has data and a living schema. That creates a *new* empty Alembic history, loses
your prior revision chain, and ``flask db migrate`` will autogenerate a single huge revision
that is easy to get wrong (3726, 5074). Commit ``migrations/`` to git and use
``flask db upgrade`` on each environment.

Fix options
-----------

A) Automatic on app startup (production / ``flask run``)
   On normal worker startup (when ``ESTITHMAR_SKIP_STARTUP_MIGRATE`` is not set), the app runs
   ``prepare_mssql_legacy_invoice_fks`` (MSSQL only, idempotent) **before** ``flask_migrate_upgrade()``.
   No manual step is required for deploys. Opt out of the prepare step only with::

       ESTITHMAR_SKIP_MSSQL_PREPARE_ON_STARTUP=1

   It never runs ``DROP TABLE`` or deletes rows (members, agents, contributions/payments stay intact).

B) Optional manual CLI (troubleshooting)
   From the project root with venv active and ``FLASK_APP`` set::

       flask prepare-mssql-legacy-fks

   Then ``flask db upgrade`` if you are not relying on startup migrate. Same FK + ``DF_*`` drops as (A).

C) Edit your Alembic revision (e.g. ``migrations/versions/53c9b541c9f4_.py``)

   At the **top** of ``upgrade()`` (before any ``op.drop_table('invoices')``), add::

       from estithmar.mssql_alembic import alembic_drop_foreign_keys_referencing_tables

       def upgrade():
           alembic_drop_foreign_keys_referencing_tables(op, {"invoice_lines", "invoices"})
           # ... rest of autogenerated upgrade()

   If autogenerate also removes tables like ``role_permissions`` / ``user_extra_permissions``,
   extend the set or run the same pattern *before* dropping those tables.

D) Partial upgrade / wrong revision state

   If upgrade failed midway, MSSQL usually rolls back the transaction and ``alembic_version``
   is unchanged — edit the revision as in (C) and run ``flask db upgrade`` again.

   If someone stamped the DB incorrectly, fix ``alembic_version`` only with DBA review.

Error 3728 — ``FK_contributions_invoice`` is not a constraint
-------------------------------------------------------------
The autogenerated revision often includes ``batch_op.drop_constraint('FK_contributions_invoice')``
even when that FK was already removed (or never existed under that exact name). SQL Server then
returns **3728**.

**Fix (pick one):**

1. Restart the app (startup runs MSSQL prepare + migrate automatically), or run
   ``flask prepare-mssql-legacy-fks`` manually — both use ``sys.*`` and drop
   ``FK_contributions_invoice`` only if present.

2. Edit ``migrations/versions/53c9b541c9f4_.py``: **before** the ``with op.batch_alter_table('contributions'...)``
   block that drops ``FK_contributions_invoice``, insert::

       from estithmar.mssql_alembic import alembic_mssql_drop_fk_if_exists

       alembic_mssql_drop_fk_if_exists(op, table="contributions", constraint_name="FK_contributions_invoice")

   Then **remove** or comment out the corresponding ``batch_op.drop_constraint('FK_contributions_invoice')``
   line inside ``batch_alter_table`` (so Alembic does not try to drop it twice).

Error 5074 / 4922 — default constraint blocks ``ALTER COLUMN``
----------------------------------------------------------------
Example: ``The object 'DF_member_documents_uploaded_at' is dependent on column 'uploaded_at'``.

Autogenerate often issues ``batch_op.alter_column`` for ``DATETIME2`` → ``DateTime`` without
dropping the table's ``DF_*`` default first. The revision ``migrations/versions/66eb2407200d_.py``
in this repo does the safe order. Startup (see (A) above) runs the same ``DF_*`` cleanup before
migrate; manual ``flask prepare-mssql-legacy-fks`` is optional. No row deletes.

``flask db migrate`` / "Target database is not up to date"
---------------------------------------------------------
If ``flask db migrate`` says the database is not up to date, run ``flask db upgrade`` first (after
fixing any failing revision). Do **not** generate a second autogenerate revision on top of a
half-applied database without resolving the current head.

Hang on ``flask prepare-mssql-legacy-fks`` / no output for minutes
-------------------------------------------------------------------
Previously ``create_app()`` still queried the DB (project codes, ``ensure_app_schema``, seed)
before the CLI ran — if SQL Server was slow or unreachable, the process appeared frozen.

That path is skipped for ``prepare-mssql-legacy-fks`` and ``flask db …``. You should see a line
like ``prepare-mssql-legacy-fks: running …`` immediately, then either success or an error within
about ``ESTITHMAR_MSSQL_LOGIN_TIMEOUT`` seconds (default **30**). Increase or lower via ``.env``.

Startup behaviour (important)
-----------------------------
The app **no longer** runs ``flask db upgrade`` automatically while you execute ``flask db …`` or
``flask prepare-mssql-legacy-fks`` (that used to recurse and hide problems). Full DB startup
(migrate, ``ensure_app_schema``, seeding) runs on normal ``flask run`` / production workers only,
not during those CLI commands.

To skip startup migrations entirely, set::

    ESTITHMAR_SKIP_STARTUP_MIGRATE=1
